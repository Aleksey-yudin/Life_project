# Backend Technical Documentation & Requirements

## 1. Архитектура и платформа

- Основной облачный бэкенд — [Supabase](https://supabase.com/)
- Использовать имеющиеся в Supabase сервисы:
    - База данных (PostgreSQL)
    - Supabase Auth для аутентификации пользователей
    - Supabase Storage (опционально, для файлов/аватарок)
    - Auto-generated REST и GraphQL API
- Возможность в будущем разместить собственный бэкенд на Node.js (Express/Fastify/Nest.js) или Python (FastAPI), если потребуется.

---

## 2. Структура данных и доступ

- Все пользовательские данные (бюджет, привычки, задачи) хранятся в отдельных таблицах в PostgreSQL (Supabase Database).
- Минимальная структура:
    - **users** — пользователи (основная информация, email, id)
    - **budgets** — финансовые транзакции, кошельки
    - **categories** — категории доходов/расходов
    - **habits** — привычки
    - **habit_entries** — отметки в календаре по каждой привычке
    - **todos** — задачи, подзадачи
- У каждой таблицы должен быть внешний ключ `user_id` для разграничения доступа.
- Доступ к данным реализуется через [Row-Level Security (RLS)](https://supabase.com/docs/guides/auth/row-level-security) — только свои данные для каждого пользователя.

---

## 3. API

- Использовать auto-generated REST API Supabase для чтения и записи данных с фронтенда через JS SDK.
- Для сложных операций можно использовать [Supabase Edge Functions](https://supabase.com/docs/guides/functions) (на Node.js/Deno).
- Все фронтенд-запросы должны быть защищены токеном доступа пользователя (JWT), полученным через Supabase Auth.
- Для более гибких сценариев в будущем предусмотреть возможность подключения собственного кастомного API (REST/GraphQL).

---

## 4. Аутентификация и безопасность

- Использовать Supabase Auth: email+password, хэширование и безопасное хранение пароля обеспечивается сервисом.
- JWT-токены для авторизации запросов.
- Использовать политики Row-Level Security (RLS) для всех таблиц — только свои записи.
- Доступ к Storage — только для своих файлов (если понадобится).

---

## 5. Мониторинг, резервное копирование, миграции

- Для небольших личных и MVP-проектов Supabase обеспечивает автоснапшоты и бэкапы БД.
- Для миграций — использовать встроенный SQL Editor и миграции Supabase CLI.
- Логи и мониторинг запроса доступны из админ-панели Supabase.

---

## 6. Масштабирование и развитие

- Архитектура проектируется с учётом будущего перехода на собственный сервер (если проект вырастет).
- При необходимости возможно разделение на микросервисы и вынесение бизнес-логики в отдельные функции или серверные приложения.
- Все данные должны быть независимы от платформы хранения, чтобы упростить миграцию (стандартизированные таблицы, ясные схемы данных).

---

## 7. Интеграции

- Возможна интеграция с внешними API (почта, уведомления, платёжные системы — в будущем).
- Поддержка Webhooks и Edge Functions для обработки событий.

---

## 8. Security Best Practices

- Минимизировать публикацию приватных ключей (все чувствительные данные — через .env).
- Всегда использовать HTTPS для клиент-серверных запросов.
- Не держать открытых CORS для всех доменов, только нужные.
- Регулярно проверять и тестировать RLS-политики.
- Использовать уникальные id (uuid) для всех записей.
