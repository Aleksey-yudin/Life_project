# Инструкция по тестированию исправлений

## Проблема
При сохранении сущностей (кошельки, категории, операции, привычки, задачи) ничего не происходит - окно не закрывается и данные не добавляются. Ошибки только логируются в консоль, но не показываются пользователю.

## Решение
Добавлены видимые уведомления об ошибках и успешных операциях (Snackbar/Alert) во все компоненты, выполняющие операции сохранения.

## Что было исправлено

### 1. Бюджет (Budget)
- **WalletList** - добавлены Snackbar для ошибок и успеха при добавлении/редактировании/удалении кошельков
- **CategoryList** - добавлены Snackbar для ошибок и успеха при работе с категориями
- **TransactionDialog** - добавлены Snackbar для ошибок и успеха при добавлении операций

### 2. Привычки (Habits)
- **HabitCalendar** - добавлены Snackbar для ошибок и успеха при отметке привычек
- **MoodCalendar** - добавлены Snackbar для ошибок и успеха при сохранении настроения

### 3. Задачи (Todo)
- **TodoPage** - добавлены Snackbar для ошибок и успеха при добавлении задач
- **TodoCalendar** - добавлены Snackbar для ошибок и успеха при изменении статуса задач
- **TodoStore** - исправлен `toggleTodoStatus` для возврата значения и обработки ошибок

## Как тестировать

### Предварительные требования
1. Убедитесь, что Supabase проект настроен и запущен
2. В файле `frontend/.env.local` должны быть указаны корректные ключи:
   ```
   NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
   ```
3. База данных должна содержать все необходимые таблицы (см. `supabase-migrations/002_clean_schema.sql`)
4. RLS политики должны быть включены и настроены

### Тестовые сценарии

#### 1. Авторизация
- Откройте `/login`
- Введите email и пароль (должен существовать пользователь в Supabase Auth)
- При успешном входе - редирект на `/dashboard`
- При ошибке - появится красное уведомление с текстом ошибки

#### 2. Кошельки (Бюджет → Кошельки)
- Нажмите "Добавить"
- Заполните название и баланс
- Нажмите "Сохранить"
- **Ожидаемый результат**: 
  - При успехе: зелёное уведомление "Кошелёк успешно добавлен", диалог закрывается, кошелёк появляется в списке
  - При ошибке: красное уведомление с текстом ошибки (например, "User not authenticated" если не авторизован)
- Попробуйте редактировать и удалить кошелёк - должны быть аналогичные уведомления

#### 3. Категории (Бюджет → Категории)
- Аналогично кошелькам: добавление, редактирование, удаление
- Проверьте, что уведомления появляются

#### 4. Операции (Бюджет → Добавить операцию)
- Откройте диалог добавления операции
- Заполните все поля (кошелёк, категория, сумма, дата)
- Нажмите "Сохранить"
- **Ожидаемый результат**: уведомление об успехе или ошибке

#### 5. Привычки (Привычки)
- Создайте новую привычку (если есть интерфейс)
- На календаре привычки кликните правой кнопкой на день
- Выберите статус (Выполнено/Частично/Пропущено)
- **Ожидаемый результат**: уведомление об успешном сохранении или ошибке
- Проверьте календарь настроения - кликните на день, выберите настроение, сохраните

#### 6. Задачи (Задачи)
- Нажмите "Добавить задачу"
- Заполните форму (обязательно название)
- Нажмите "Добавить"
- **Ожидаемый результат**: уведомление об успехе или ошибке
- В календаре задач кликните на чекбокс задачи для изменения статуса
- Должно появиться уведомление

## Что проверить дополнительно

### Консоль браузера
- Откройте DevTools (F12) → Console
- При операциях должны появляться логи:
  - `Error saving wallet: ...` (если есть ошибка)
  - `Error adding transaction: ...` и т.д.
- Но теперь ошибки также показываются в UI

### Network запросы
- Во вкладке Network должны отправляться запросы к Supabase API
- Проверьте статус ответов:
  - 200-299 - успех
  - 400-499 - ошибки клиента (например, RLS violation)
  - 500+ - ошибки сервера

### RLS политики
Если появляются ошибки вида "row-level security policy violation":
1. Зайдите в Supabase Dashboard
2. Перейдите в Authentication → Policies
3. Проверьте, что для всех таблиц включен RLS
4. Проверьте политики - они должны разрешать операции только для `auth.uid() = user_id`

### Авторизация
Если появляется "User not authenticated":
1. Проверьте, что пользователь вошёл в систему
2. Проверьте, что сессия активна (Supabase Auth)
3. Проверьте, что `user.id` передаётся корректно во все операции

## Известные ограничения

1. **Авторизация**: Текущая реализация использует Supabase Auth. Необходимо создать пользователя в Supabase Auth перед входом.
2. **RLS**: Все таблицы должны иметь RLS политики, иначе запросы будут падать с ошибкой доступа.
3. **Офлайн-режим**: Не поддерживается, требуется постоянное подключение к интернету.

## Устранение типовых проблем

### Проблема: Ничего не происходит при нажатии кнопки "Сохранить"
**Решение**: 
1. Откройте консоль браузера (F12)
2. Проверьте, есть ли ошибки
3. Проверьте, появилось ли уведомление об ошибке (Snackbar внизу экрана)
4. Проверьте Network вкладку - отправляются ли запросы

### Проблема: "User not authenticated"
**Решение**:
1. Убедитесь, что вы вошли в систему
2. Проверьте, что `useAuthStore()` возвращает `user` объект
3. Перезайдите в систему

### Проблема: "RLS policy violation"
**Решение**:
1. Выполните миграцию из `supabase-migrations/002_clean_schema.sql`
2. Убедитесь, что RLS включен для всех таблиц
3. Проверьте, что во все записи добавляется корректный `user_id`

### Проблема: Сess expired или invalid session
**Решение**:
1. Выйдите из системы
2. Очистите кэш браузера (или localStorage/sessionStorage)
3. Войдите заново

## Мониторинг

После исправлений все операции должны:
1. **Показывать визуальную обратную связь** (уведомления об успехе/ошибке)
2. **Логировать ошибки в консоль** для отладки
3. **Корректно обновлять UI** после успешных операций
4. **Закрывать диалоги** только при успешном сохранении

## Контакты для поддержки

Если проблемы остаются:
1. Проверьте логи консоли браузера
2. Проверьте Network вкладку (статус запросов)
3. Проверьте Supabase Dashboard → Logs
4. Убедитесь, что все миграции выполнены
